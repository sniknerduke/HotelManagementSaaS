## 📋 Step-by-Step Implementation Roadmap

> Sequential order — **do Step 1 first, then Step 2, and so on**. Each step builds on the previous one.
> Benchmarked against [the-wild-oasis](https://github.com/sudeepmahato16/the-wild-oasis), [Hotel-Booking-MERN](https://github.com/arnobt78/Hotel-Booking-Management-System--React-MERN-FullStack), [jsanhuo/HotelManagementSystem](https://github.com/jsanhuo/HotelManagementSystem), and industry HMS standards.

```mermaid
graph LR
    S1["Step 1\nSeed Data"] --> S2["Step 2\nComplete CRUD"] --> S3["Step 3\nSecurity"] --> S4["Step 4\nService Links"] --> S5["Step 5\nFrontend Wiring"]
    S5 --> S6["Step 6\nValidation"] --> S7["Step 7\nAnalytics"] --> S8["Step 8\nHousekeeping"] --> S9["Step 9\nTesting"] --> S10["Step 10\nDevOps"]
```

### Step 2 — 🔧 Complete All CRUD Operations *(Backend Core)*

> **Why second?** The backend can only create a few things right now. Need full Create-Read-Update-Delete for every entity before anything else works.

#### 2a. Room & Room Type CRUD (`InventoryResource.java`)
- [x] `GET /api/inventory/rooms` — list all rooms (with filters by status, type, floor)
- [x] `GET /api/inventory/rooms/{id}` — get single room
- [x] `PUT /api/inventory/rooms/{id}` — update room (number, type, status, floor)
- [x] `DELETE /api/inventory/rooms/{id}` — soft-delete or deactivate room
- [x] `PATCH /api/inventory/rooms/{id}/status` — change room status (for front desk / housekeeping)
- [x] `GET /api/inventory/room-types` — list all room types
- [x] `PUT /api/inventory/room-types/{id}` — update room type (name, price, capacity, description, image)
- [x] `DELETE /api/inventory/room-types/{id}` — delete room type (only if no rooms attached)

#### 2b. Booking CRUD (`BookingResource.java`)
- [x] `GET /api/bookings/{id}` — get single booking with all details
- [x] `PUT /api/bookings/{id}` — update booking (dates, room, guest count, notes)
- [x] `DELETE /api/bookings/{id}` — cancel booking (set status to CANCELLED)
- [x] `POST /api/bookings/{id}/check-in` — check guest in → status=`CHECKED_IN`, room→`OCCUPIED`
- [x] `POST /api/bookings/{id}/check-out` — check guest out → status=`CHECKED_OUT`, room→`CLEANING`

#### 2c. User CRUD (`UserResource.java`)
- [x] `PUT /api/users/{id}` — update profile (name, phone, nationality, address, avatar)
- [x] `DELETE /api/users/{id}` — deactivate account (soft delete)
- [x] `PATCH /api/users/{id}/role` — admin changes role (GUEST ↔ STAFF ↔ ADMIN)
- [x] `PUT /api/users/{id}/password` — change password (requires current password)

#### 2d. Payment CRUD (`PaymentResource.java`)
- [x] `GET /api/payments` — list all payments (admin, with filters by status, date range)
- [x] `GET /api/payments/{id}` — get single payment details
- [x] `POST /api/payments/{id}/refund` — refund payment (partial or full)

### Step 3 — 🔐 Security & Authentication *(Lock It Down)*

> **Why third?** CRUD must work first so you have endpoints to protect. Now add auth so only the right people can access them.

- [x] Add BCrypt dependency to `user-service/pom.xml`
- [x] Hash passwords in `UserResource.register()` using BCrypt
- [x] Verify hashed password in `UserResource.login()` using BCrypt
- [x] Enable `smallrye-jwt` in `application.properties`
- [x] Generate RSA key pair (`privateKey.pem` + `publicKey.pem`) for JWT signing
- [x] Create `JwtService.java` — generates JWT on login with userId, email, role claims
- [x] Return JWT token in login response body
- [x] Update `SocialAuthResource.java` — return real JWT instead of mock token string
- [x] Add `@RolesAllowed("ADMIN")` guards on admin-only endpoints (delete user, change role, list all payments, etc.)
- [x] Add `@RolesAllowed("GUEST")` or `@Authenticated` on user-specific endpoints (update own profile, view own bookings)
- [x] Add `POST /api/users/logout` — frontend clears token, backend can optionally blacklist
- [x] Update `AuthContext.tsx` — store JWT in `localStorage`, decode role from token, auto-attach to `api/client.ts`
- [x] Add frontend route guards — redirect to `/login` if no token, redirect from `/admin` if not ADMIN role

### Step 4 — 🔗 Service-to-Service Communication *(Connect the Microservices)*

> **Why fourth?** With CRUD + auth done, now make the services talk to each other for real business logic.

- [x] Add Quarkus REST Client dependency to `booking-service/pom.xml`
- [x] Create `InventoryClient.java` interface — call `GET /api/inventory/rooms/{id}` to check availability
- [x] Update `BookingResource.createBooking()` — call inventory to verify room is `AVAILABLE`, then set room to `OCCUPIED`
- [x] Update `BookingResource.check-out()` — call inventory to set room back to `CLEANING`
- [x] Add Quarkus REST Client to `payment-service/pom.xml`
- [x] Create `BookingClient.java` interface — call `PATCH /api/bookings/{id}/status`
- [x] Update `PaymentResource.createPayment()` — after payment succeeds, call booking service to set status to `CONFIRMED`

### Step 6 — ✅ Input Validation & Data Quality

> **Why sixth?** Now that everything works end-to-end, harden it so bad data can't get in.

- [ ] Add `@NotBlank`, `@Email`, `@Size` on all DTO record fields across all services
- [ ] Add `@Min(1)` on numeric fields (guest count, prices)
- [ ] Add date validation — check-in must be before check-out, can't book in the past
- [ ] Create `GlobalExceptionMapper.java` in each service — catch `ConstraintViolationException` → return clean JSON errors
- [ ] Add pagination to all list endpoints — `?page=0&size=20&sort=createdAt` using Panache `.page()` / `.range()`
- [ ] Add unique constraints and proper indexes to all entities
- [ ] Frontend form validation — match backend rules, show inline errors before submit
